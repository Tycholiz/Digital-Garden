Orchestrators

Orchestrators

  • Tools to manage, scale, and maintain containerized applications
  • ie. Kubernetes, Docker Swarm
  • Orchestration tools are designed to handle Docker containers running stateless applications
    • as such, you should not run stateful applications in orchestration tools which are built for stateless apps.
    • this makes sense, seeing as containers can be killed and restarted without any issue, multiple can run at once etc.
      • The same cannot be said for a database, which is why we typically don't dockerize DBs

In general, the development workflow looks like this:

  1. Create and test individual containers for each component of your application by first creating Docker images.
  2. Assemble your containers and supporting infrastructure into a complete application, expressed either as a Docker stack file or in Kubernetes YAML.
  3. Test, share and deploy your complete containerized application.

Kubernetes vs Docker Swarm mode

In Kubernetes, an application can be deployed using a combination of pods, deployments, and services (or micro-services).

Whereas, in Docker Swarm, applications can be deployed as services (or micro-services) in a Swarm cluster. YAML files can be used to specify multi-container. Moreover, Docker Compose can deploy the app.

Kubernetes

All containers in Kubernetes are scheduled as pods

  • Kubernetes uses client-server architecture

pod

  • def: one or more containers deployed together on one host, thereby sharing the same resources (of the host)
    • ex. if we have 5 containers of a mongodb service deployed, and 3 of them were on the same host (ex. same machine), those 3 together would be called a Pod
    • pod refers to a pod of whales or pea pod
  • they often share storage/network and namespaces
  • they are mortal. They are born and when they die, they are not resurrected.
  • Individual Pods are not intended to run multiple instances of the same application,
  • in a pre-container world, being executed on the same physical or virtual machine would mean being executed on the same logical host.
    • logical host would contain relatively tightly coupled code
  • Containers within a Pod share an IP address and port space, and can find each other via localhost.

Deployment

  • def: scalable groups of pods
  • A Deployment provides declarative updates for Pods and ReplicaSets. -You describe a desired state in a Deployment, and the Deployment Controller changes the actual state to the desired state at a controlled rate

Service

  • def: An abstract way to expose an application running on a set of Pods as a network service.
  • can be thought of as a logical set of pods and a policy by which to access them
    • the front end shouldn't care which backend it uses. Services allow us to achieve this decoupling
  • since pods of a single service can exist on different machines, it makes sense for us to be able to interact with the service itself, so that we can orchestrate activities between all containers that are part of a service(?)
